The `services.miloapis.com` API is Datum Cloud's service catalog. A **Service** is the identity record for a managed capability offered on the platform. A consumer opts in by creating a **ServiceEntitlement** in their own Project, and the platform reflects each active or pending entitlement back to the provider as a **ServiceConsumer**. You create the resources you own with `datumctl apply -f` and inspect the rest with `datumctl get` and `datumctl describe`.

<Warning>
  The Services API is **alpha** (`v1alpha1`). Fields and behavior may change in backward-incompatible ways between releases.
</Warning>

## How the pieces fit together

- A **Service** is published to the catalog and identified by its canonical `spec.serviceName`. Downstream resources reference a service by that name.
- A **ServiceEntitlement** is written into the consumer Project to request use of a service; the services operator reconciles it into the provider Project.
- A **ServiceConsumer** is created by the platform in the provider Project for every active or pending entitlement. Providers do not create these directly — they approve or deny requests by writing the entitlement's approval decision.

<Tip>
  Inspect the exact fields for any resource with `datumctl explain`, for example `datumctl explain services.services.miloapis.com` or `datumctl explain serviceentitlements.services.miloapis.com`.
</Tip>
